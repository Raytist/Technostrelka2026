# Technical Design Document: Сервис мониторинга подписок (Yandex Mail / Hackathon Edition)

## 1. Архитектурный сдвиг: Переход на парсинг почты Яндекса
В связи с отказом от API ZenMoney источником данных об оплатах становятся почтовые ящики Яндекса пользователей (электронные чеки от ОФД и письма от сервисов).
**Ключевые изменения в архитектуре:**
1. **Асинхронный сбор данных:** Подключение к Яндекс Почте по протоколу IMAP через OAuth авторизацию Яндекса.
2. **Движок парсинга (Parsing Engine):** Извлечение датапоинтов (Сумма, Валюта, Мерчант, Дата) из HTML-чеков и писем.
3. **Smart Discovery (Умный поиск):** Алгоритм автоматически находит регулярные платежи. Все "подозрительные" новые подписки падают пользователю в ленту "На подтверждение" (нужно принять или отклонить).
4. **Уведомления:** Firebase Cloud Messaging (FCM) для мобильных Push-уведомлений в приложении (предупреждения за 1 день до списания и алерты о новых найденных подписках).
5. **Отказоустойчивая синхронизация:** Инкрементальный pull писем (Polling) раз в 6 часов фоновыми воркерами.

---

## 2. Памятка для Frontend-разработчиков (От Backend) 💡
Для максимально быстрой и безболезненной интеграции:
* **Авторизация:** JWT-токен (`access_token`) нужно прикреплять к каждому запросу (кроме логина) в заголовках: `Authorization: Bearer <token>`.
* **Flow подписок:** Главный экран (Дашборд) должен дёргать `/dashboard/summary` для красивой цифры сверху, а ниже выводить активные подписки из `/subscriptions?status=active`.
* **Уведомление (Лента "На проверку"):** На главном экране сделайте иконку с бейджиком – это массив из `/dashboard/feed`. Если массив не пустой, фронт должен отрендерить карточки (Tinder-свайпами влево/вправо), которые вызывают `/verify` (Да) или `/reject` (Нет).
* **Синхронизация:** Опрос почты занимает время. Когда юзер жмет "Обновить письма", дёргайте `POST /sync/start`, показывайте лоадер/тост "Ищем подписки...", и через пару секунд можете пере-запрашивать `/dashboard/feed`.

## 3. Список эндпоинтов (API Endpoints) с примерами данных

Все эндпоинты (кроме `/api/v1/auth/yandex` и `../callback`) требуют заголовок `Authorization: Bearer <ваш_токен>`.

### **Auth (Авторизация через Яндекс)**
| Метод | Эндпоинт | Описание |
|---|---|---|
| `GET` | `/api/v1/auth/yandex` | Получение ссылки для редиректа юзера в браузере на Яндекс OAuth. |
| `GET` | `/api/v1/auth/yandex/callback` | Коллбэк для OAuth-авторизации (сохранение токенов). |

**Пример ответа `GET /auth/yandex/callback?code=...`:**
```json
{
  "access_token": "eyJhbGciOiJI...",
  "token_type": "bearer",
  "email": "Savelijfamilia130@yandex.ru"
}
```

### **User (Профиль)**
| Метод | Эндпоинт | Описание |
|---|---|---|
| `GET` | `/api/v1/users/me` | Данные профиля (включая флаг `push_enabled`). |
| `PATCH` | `/api/v1/users/me` | Обновление настроек (токен FCM / пуши). |

**Пример Body для `PATCH /users/me`:**
```json
{
  "push_enabled": false,
  "fcm_token": "firebase_device_identifier_token_here"
}
```

### **Dashboard / Stats (Аналитика)**
| Метод | Эндпоинт | Описание |
|---|---|---|
| `GET` | `/api/v1/dashboard/summary` | Агрегация трат: сумма активных подписок в месяц, прогноз на 30 дней. |
| `GET` | `/api/v1/dashboard/feed` | Лента **отфильтрованных транзакций**, которые ждут апрува (Новые подписки из парсера). |

**Пример ответа `GET /dashboard/summary`:**
```json
{
  "monthly_total": 1500.00,
  "active_subscriptions_count": 3,
  "unverified_count": 2
}
```

**Пример ответа `GET /dashboard/feed`:**
```json
[
  {
    "id": "c7a8e...",
    "receipt_date": "2026-03-18T10:00:00",
    "merchant_name": "ООО Амедиатека",
    "amount": 599.0,
    "is_trial": true
  }
]
```

### **Subscriptions (Подписки & Валидация)**
Отдельный процесс: Подтверждение найденных алгоритмом подписок пользователем.
| Метод | Эндпоинт | Описание |
|---|---|---|
| `GET` | `/api/v1/subscriptions` | Список подписок (фильтры: `?status=active`, `canceled`, `trial`, `unverified`). |
| `POST` | `/api/v1/subscriptions` | **Ручное добавление подписки (Custom)** пользователем. |
| `PATCH` | `/api/v1/subscriptions/{id}/verify` | Пользователь нажал "Да, это подписка". |
| `PATCH` | `/api/v1/subscriptions/{id}/reject` | Пользователь нажал "Нет, это просто покупка". |
| `PATCH` | `/api/v1/subscriptions/{id}` | Изменение параметров (переименование/дата). |

**Пример Body для `POST /subscriptions`:**
```json
{
  "name": "Gym Membership",
  "amount": 1500.0,
  "periodicity": "monthly",
  "next_payment_date": "2026-04-01T12:00:00Z"
}
```

**Пример ответа `GET /subscriptions`:**
```json
[
  {
    "id": "abc...",
    "raw_merchant_name": "ООО ЯНДЕКС",
    "name": "Яндекс Плюс (Trial?)",
    "amount": 1.00,
    "status": "unverified",
    "periodicity": "monthly",
    "next_payment_date": null
  }
]
```

### **Sync (Синхронизация)**
| Метод | Эндпоинт | Описание |
|---|---|---|
| `POST` | `/api/v1/sync/start` | Ручной запуск фонового `MailSyncWorker` для обновления писем. Возвращает `{status: "Sync queued successfully"}`. |

### **Таблица: Subscriptions**

|---|---|---|

## 5. Алгоритм работы пайплайна (Smart Rules-Free Logic)

### **Этап 1. Ingestion (Сбор писем)**
Сервер проверяет наличие новых писем по IMAP каждые 6 часов (или по кнопке ручной синхронизации), обрабатывая последние свежие `UNSEEN` письма.

### **Этап 2. Parsing Engine (Извлечение чеков)**
Письма прогоняются через NLP-парсер (BeautifulSoup + Regex). Извлекаются сырые данные, а также флаги `is_trial` (если найдены фразы типа *первый месяц*, *бесплатно*).

### **Этап 3. Smart Recurrence Scanner (Поиск подписок)**
1. Если парсер видит **2+ чека** с нормальным интервалом от одного ИНН/названия — он генерирует `unverified` подписку и ждет свайпа от юзера.
2. Если парсер видит **новый Одиночный чек**, он идет в Sanity-фильтр:
   - **Trial Payment:** Если `is_trial=True` ИЛИ сумма чека: `0.5 руб - 15.0 руб`. Добавляется в Feed с пометкой *"Пробный период (Trial)?"*.
   - **Обычный чек (>15 руб):** Сумма должна быть круглой по подпискам (оканчиваться на 9 или 0, напр. 99, 290). 
   - **Отсев мусора:** "Пятерочка", "Магнит", "Аптека", "Wildberries", "Ozon" жестко блокируются.
   - Чек попадает в `Dashboard Feed`!

### **Этап 4. User Validation (Человеческий фильтр)**
Если подписку нашел алгоритм — она падает во фронтенд `GET /api/v1/dashboard/feed`. Пользователь видит карточку ОФД чека, и дальше 2 сценария:
1. **Подтвердить** (`/verify`) - подписка становится активной `active`.
2. **Отклонить** (`/reject`) - подписка помечается `rejected` и игнорируется алгоритмами навсегда.

## 6. Инфраструктура фоновых задач (Celery + RabbitMQ)
1. **`periodic_mail_sync`:** Глобальный сборщик новых писем всех пользователей раз в 6 часов.
2. **`mail_fetch_task`:** Персональное чтение почты конкретного юзера по IMAP.
3. **`receipt_parse_task`:** Индивидуальный NLP анализ каждого нового чека с проставлением `is_trial` флагов.
4. **`subscriptions_matcher_task`:** Сканатор, который переключает базу и ищет зацепки на оформление новой подписки.
5. **`firebase_notifier_task`:** Ежедневный пушер FCM-предупреждений ("До списания остался 1 день") для `/users/me:push_enabled==True`.
